# Task tracker API

## Запуск и использование

```bash
docker compose run --rm --service-ports app bash -c "bin/setup --skip-server && bin/dev -b 0.0.0.0"
```

Сервер доступен по адресу [localhost:3000](http://localhost:3000/).

[OpenAPI схема](swagger/v1/swagger.yaml).

Попробовать API в действии можно через интерактивный интерфейс Swagger UI: [localhost:3000/api-docs](http://localhost:3000/api-docs).

## Стек

- Ruby 4.0
- Rails 8.1
- PostgreSQL 16
- Docker + Docker Compose
- RSpec + Rswag

## Реализованная функциональность

### 1. Управление задачами

- Базовый CRUD:
  - `GET /api/v1/tasks` - просмотр списка задач с фильтрацией/поиском по дате внутри временного промежутка \[`start_date`, `end_date`\] и по статусу `status`.
  - `POST /api/v1/tasks` - создание новой задачи.
  - `GET /api/v1/tasks/:id` - просмотр детальной информации о задаче.
  - `PATCH /api/v1/tasks/:id` - редактирование задачи.
  - `DELETE /api/v1/tasks/:id` - удаление задачи.

- Атрибуты задачи:
  - `title` - заголовок/название задачи.
  - `description` - дополнительное описание задачи.
  - `scheduled_date` - дата, на которую запланировано выполнение задачи.
  - `status` - статус задачи. В текущей реализации `status` имеет 2 значения: `scheduled` - задача запланирована, `completed` - задача выполнена. Набор статусов может быть расширен.

### 2. Управление тегами задач

- Базовый CRUD:
  - `GET /api/v1/tags` - просмотр списка тегов.
  - `POST /api/v1/tags` - создание нового тега.
  - `GET /api/v1/tags/:id` - просмотр детальной информации о теге.
  - `PATCH /api/v1/tags/:id` - редактирование тега.
  - `DELETE /api/v1/tags/:id` - удаление тега.

- Атрибуты тега:
   - `name` - уникальное имя/название тега.

- Задача может иметь несколько тегов. Один и тот же тег могут иметь несколько задач.

- Теги связываются с задачей при создании или редактировании задачи путём указания идентификаторов тегов `tag_ids`.

- Существуют системные теги, удаление и изменение которых недоступно с помощью API. Изначально в приложении уже имеются системные теги «отчетность», «операции» и «звонок».

### 3. Повторяющиеся/периодические задачи

- Для задач существует возможность создать правило повтора:
  - `once` - без повторения (одиночная задача).
  - `daily` - каждый n-й день (1 – ежедневная, 2 – раз в два дня, 7 – еженедельная, и т. д.).
  - `monthly_specific_date` - каждое определенное число месяца (от 1 до 31).
  - `specific_dates` - в указанные даты.
  - `odd_even_days` - по четным/нечетным дням месяца.

- Правила повторения задаются с помощью дополнительных атрибутов
  - `schedule_type` – тип повтора.
  - `schedule_options` – опции повтора.
  - `active` – активно ли правило повтора.

- Повторяемость задач реализована с помощью шаблонов, экземпляров и виртуальных задач.
  - Шаблоны хранят заголовок задачи, её описание и правила повторения.
  - Экземпляры хранят информацию о статусе задачи на конкретный день. Экземпляры сохраняются только при смене статуса задачи.
  - Виртуальные задачи, в отличие от экземпляров, не имеют записи в БД - они формируются «на лету».
  - При запросе списка задач происходит поиск существующих экземпляров, а также на основе правил повторения из активных шаблонов вычисляются дни, для которых должны быть отображены виртуальные задачи.
  - Существование шаблонов, экземпляров и виртуальных задач закрыто от клиента API абстракцией - клиент считает, что работает с задачами - `/api/v1/tasks[/:id]`.
  - Идентификаторы задач `/:id` в API представляют собой строку, состоящую из пары - идентификатор шаблона и дата задачи (`1_1995-12-21`). Это позволяет клиенту не различать экземпляры от виртуальных задач, а серверу осуществлять удобный и быстрый поиск ввиду уникальности указанной пары и наличия соответствующего индекса в БД.
